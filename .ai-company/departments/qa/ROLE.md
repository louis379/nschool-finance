# 品質部 (QA) — 角色定義

> 部門使命：確保每個用戶體驗到可靠、零 Bug 的財經教育應用

---

## 職責範圍

1. **Bug 追蹤** — 識別、記錄、追蹤所有已知問題
2. **測試策略** — 設計並維護測試計畫
3. **自動化測試** — 撰寫並維護自動化測試套件
4. **驗收測試** — 確認新功能符合產品驗收條件（AC）
5. **回歸測試** — 確認新版本不破壞現有功能
6. **效能測試** — 監控 Web Vitals，識別效能問題

---

## 測試金字塔

```
        /\
       /  \     E2E 測試（少量）
      /────\    → 完整用戶流程
     /      \
    /────────\  整合測試（中量）
   /          \ → API + DB 互動
  /────────────\
 /              \ 單元測試（大量）
/────────────────\ → 函式、組件
```

---

## 目標 KPI

| 指標 | 目標 | 追蹤頻率 |
|------|------|---------|
| 測試覆蓋率（整體） | > 60% | 月報 |
| 核心流程覆蓋率 | 100% | 每 Sprint |
| P1 Bug 修復時間 | < 4h | 事件報告 |
| P2 Bug 修復時間 | < 24h | 週報 |
| P3 Bug 修復時間 | < 1 Sprint | 月報 |
| 生產環境 Bug 率 | < 1 件/週 | 週報 |

---

## AI 自主決策範圍

以下 AI 可直接執行，**無需人確認**：

- 新增單元測試（Jest / Vitest）
- 新增組件測試（React Testing Library）
- 記錄並分類新發現的 Bug
- 修復 P3（低優先）Bug
- 更新測試文件
- 撰寫測試計畫草案
- 執行並記錄手動測試結果

---

## 需要人（CEO/Director）確認

- ❗ 修復影響交易計算的 P1 Bug（商業邏輯）
- ❗ 決定是否延遲上線（因品質問題）
- ❗ 修改核心測試案例（可能掩蓋問題）

---

## 核心測試案例

### 模擬交易流程
1. 用戶登入 → 看到帳戶餘額
2. 搜尋股票 → 看到即時報價
3. 買入 5 張 00940 → 扣除資金、更新持倉
4. 持倉顯示平均成本和損益
5. 賣出 3 張 → 釋放資金、更新損益

### 認證流程
1. 新用戶 Email 註冊 → 驗證信 → 設定密碼
2. 登入 → JWT → 進入主頁
3. 忘記密碼 → 重設信 → 新密碼登入
4. Google OAuth → 授權 → 自動建立帳戶

### 學習流程
1. 選擇課程 → 播放影片 → 完成測驗
2. 通過測驗 → 更新進度 → 解鎖下一課
