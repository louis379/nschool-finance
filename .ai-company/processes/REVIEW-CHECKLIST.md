# 人工 Review 清單 (Review Checklist)

> 當 AI 完成工作後，人（CEO/Director）進行 Review 時使用的清單。

---

## 快速 Review（5 分鐘）

適用於：小 Bug 修復、UI 調整、文件更新

- [ ] 程式碼邏輯看起來正確
- [ ] 沒有明顯的安全疑慮
- [ ] UI 截圖正常
- [ ] Commit message 清楚

---

## 標準 Review（15 分鐘）

適用於：新功能、重要修復

### 功能正確性
- [ ] 功能符合需求描述
- [ ] 邊界情況有處理（空值、錯誤、loading）
- [ ] 不影響現有功能（回歸測試通過）

### 程式碼品質
- [ ] 程式碼可讀性高，不需要注釋才能理解
- [ ] 沒有重複程式碼（DRY 原則）
- [ ] 沒有不必要的複雜度
- [ ] TypeScript 型別正確

### 安全性
- [ ] 沒有 SQL Injection 風險
- [ ] 用戶輸入有驗證
- [ ] 敏感資料沒有 log 出來
- [ ] API Key 沒有在前端暴露

### 使用者體驗
- [ ] 手機版面正常
- [ ] Loading 狀態有處理
- [ ] 錯誤訊息對用戶友善
- [ ] 不影響現有用戶流程

---

## 深度 Review（30 分鐘）

適用於：架構變更、資料庫變更、安全性相關

### 上述所有項目，加上：

### 架構
- [ ] 符合現有系統架構
- [ ] 不引入不必要的依賴
- [ ] 可維護性好（未來 AI 能理解並修改）
- [ ] 效能考量（N+1 查詢、不必要的重渲染）

### 資料庫（若有 Schema 變更）
- [ ] Migration 可回滾
- [ ] RLS Policy 正確設定
- [ ] Index 適當
- [ ] 不破壞現有資料

### 商業邏輯
- [ ] 交易計算正確（手續費、損益）
- [ ] 用戶隱私保護
- [ ] 符合台灣金融法規（不提供投資建議）

---

## 常見問題 Checklist

### AI 常見疏漏
- [ ] console.log 殘留
- [ ] TODO 留著沒處理
- [ ] 硬編碼的 URL 或 API endpoint
- [ ] 環境變數用 process.env 正確讀取（非硬編碼）
- [ ] 圖片 alt text 有填
- [ ] 非同步錯誤有 try-catch

### UI 常見問題
- [ ] 長文字有截斷處理（text-ellipsis）
- [ ] 空狀態有設計（Empty State）
- [ ] 手機鍵盤彈出不遮擋輸入框
- [ ] Dark Mode 色彩正確

---

## Review 後的動作

### 通過 ✅
```
→ 留下 LGTM 或直接 merge
→ 若有小建議，可以讓 AI 修正後直接通過
```

### 需要修改 🔄
```
→ 描述具體需要改什麼
→ 讓 AI 修改後重新 Review
```

### 拒絕 ❌
```
→ 說明拒絕原因
→ 記錄在對應 DECISIONS.md
→ 確認是否需要調整方向
```
