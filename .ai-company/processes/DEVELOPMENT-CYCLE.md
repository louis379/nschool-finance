# AI 開發循環（Development Cycle）

> 定義 AI Agent 自主迭代開發的完整流程

---

## 總覽

AI Agent 的開發循環是一個持續運作的自動化流程，每個循環包含 6 個階段。排程任務每次執行時都應遵循此流程。

```
┌─────────────┐
│  1. 搜尋趨勢 │ ← 了解外部世界的變化
└──────┬──────┘
       ▼
┌─────────────┐
│  2. 分析現狀 │ ← 讀取內部文件，了解目前狀態
└──────┬──────┘
       ▼
┌─────────────┐
│  3. 制定計畫 │ ← 決定這次循環要做什麼
└──────┬──────┘
       ▼
┌─────────────┐
│  4. 執行改進 │ ← 實際寫程式碼或更新內容
└──────┬──────┘
       ▼
┌─────────────┐
│  5. 測試驗證 │ ← 確認改動不會破壞東西
└──────┬──────┘
       ▼
┌─────────────┐
│  6. 回報紀錄 │ ← 更新所有相關文件
└──────┬──────┘
       │
       └──→ 回到步驟 1（下一個循環）
```

---

## 階段詳解

### 階段 1：搜尋趨勢

**目的：** 了解外部世界的變化，找到可以應用的新技術或設計趨勢。

**執行內容：**
- 搜尋理財 App 的最新 UX 趨勢
- 關注 Next.js / React 的新功能和最佳實踐
- 了解 FinTech 領域的產品動態
- 搜尋目標用戶群（20-35 歲台灣年輕人）的理財需求變化

**輸出：**
- 如果發現值得參考的趨勢，記錄到 `departments/product/DECISIONS.md`
- 更新 `knowledge-base/COMPETITIVE-ANALYSIS.md`（如有新發現）

### 階段 2：分析現狀

**目的：** 讀取所有內部文件，建立當前上下文。

**必讀文件（按順序）：**
1. `DAILY-STANDUP.md` — 人給的最新方向
2. `SPRINT-BOARD.md` — 當前任務和優先級
3. 對應部門的 `ROLE.md` — 自己的職責範圍
4. `processes/ESCALATION.md` — 什麼事情需要升級給人
5. `departments/qa/BUG-TRACKER.md` — 有沒有需要修的 bug

**分析項目：**
- 目前最高優先級的任務是什麼？
- 有沒有被 block 的任務？
- 有沒有可以自主完成的改善？
- 技術債是否影響開發效率？

### 階段 3：制定計畫

**目的：** 根據分析結果，決定這次循環要做什麼。

**優先級判斷：**
1. **P0 — 立即處理**：Production bug、服務中斷
2. **P1 — 本次循環**：人指定的優先任務
3. **P2 — 有空就做**：UX 微調、效能優化、技術債
4. **P3 — 排入下次**：新功能原型、探索性質

**決策規則：**
- 如果有 P0，放下所有事情先處理
- 如果有 P1 且在自主決策權限內，直接開始
- 如果 P1 需要人確認，記錄到 ESCALATION 並做 P2 的事
- 每次循環只做 1-2 個任務，確保品質

### 階段 4：執行改進

**目的：** 實際進行程式碼變更或內容更新。

**執行原則：**
- 遵循 Conventional Commits 格式
- 每個改動都要小而完整（可獨立回滾）
- 使用 TypeScript strict mode
- 遵循 Design System 的規範
- 不修改環境變數或部署設定

**自主可執行的改進類型：**
- UX/UI 微調（間距、動畫、文字）
- Bug 修復（不涉及商業邏輯）
- 效能優化（程式碼層面）
- 測試補充
- 文件更新
- 小型重構

### 階段 5：測試驗證

**目的：** 確認改動不會破壞現有功能。

**測試清單：**
- [ ] TypeScript 編譯通過（`pnpm build`）
- [ ] ESLint 檢查通過（`pnpm lint`）
- [ ] 現有測試通過（`pnpm test`）
- [ ] 手動驗證改動的功能
- [ ] 確認手機端顯示正常

**如果測試失敗：**
1. 回滾改動
2. 分析失敗原因
3. 修復問題後重新測試
4. 如果無法自行修復，記錄到 `ESCALATION.md`

### 階段 6：回報紀錄

**目的：** 更新所有相關文件，留下完整紀錄。

**必須更新的文件：**
- `SPRINT-BOARD.md` — 更新任務狀態
- `reports/CHANGELOG.md` — 記錄變更
- 對應部門的文件（如 BUG-TRACKER、TECH-DEBT 等）

**回報格式：**
```markdown
#### [時間] 循環報告

**執行任務：** [做了什麼]
**結果：** [成功/失敗/部分完成]
**改動檔案：** [列出修改的檔案]
**測試結果：** [通過/失敗]
**需要人 Review 的事：** [如果有的話]
**下次循環建議：** [接下來應該做什麼]
```

## 循環頻率

| 時段 | 頻率 | 重點 |
|------|------|------|
| 早上（6-9 AM） | 1 次 | 讀取最新方向，規劃今日工作 |
| 白天（9 AM-6 PM） | 每 3 小時 1 次 | 執行人指定的優先任務 |
| 晚上（6-10 PM） | 1 次 | 處理低風險優化，更新文件 |
| 深夜（10 PM-6 AM） | 1 次 | 自動化測試、技術債清理 |
